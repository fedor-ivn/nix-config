## Why

From the MacBook (`fedorivns-mbp`) I need to reach corporate internal
resources under `*.tcsbank.ru`, but the only machine with corp network
access is a locked-down corp laptop running Cisco Secure Client (Always-On
posture), Kaspersky, SkyGuard (DLP), and a Forcepoint/Check-Point agent.
Those agents **drop all unsolicited inbound connections** on the corp
laptop, so I cannot SSH *into* it — the obvious "expose a proxy on the corp
box" approach is dead by construction. At the same time I do **not** want my
normal personal traffic touching the corp egress (it is TLS-intercepted and
DLP-inspected); only `*.tcsbank.ru` should go that way, everything else
through my personal WireGuard.

Today this works as an ad-hoc, untracked `singbox.json` at the repo root
with WireGuard keys in cleartext and a hand-run SSH command. This change
makes the whole split-tunnel setup reproducible and declarative in the
flake, with secrets in sops.

## What Changes

- Add a **reverse-SSH SOCKS bridge**: the corp laptop dials *outbound* to
  the Mac (outbound is allowed; inbound is not) and exposes a SOCKS proxy on
  `127.0.0.1:1080` on the Mac whose traffic egresses from the corp laptop.
  Two stock-`ssh` hops (corp reverse-forwards its sshd; the Mac dials back
  with `-D` so the SOCKS egresses from corp). The corp dial is on-demand and
  self-heals via a stock-`ssh` backoff loop (so a brief Mac sleep recovers
  without re-running it); the Mac-side consumer is a nix-managed launchd `ssh`
  service supervised by `KeepAlive` (loopback-only, so continuous retry is
  fine).
- Add a **sing-box split-tunnel router** (`tun` inbound, `auto_route`) on
  `fedorivns-mbp`:
  - default → personal WireGuard endpoint;
  - `*.tcsbank.ru` → the corp SOCKS bridge;
  - private IPs → direct.
- Resolve `*.tcsbank.ru` via **FakeIP** so the corp laptop resolves the
  names remotely. This is required because the corp resolver
  (`10.219.194.5`) answers **UDP only**, and `ssh -D` carries TCP only — so
  a direct DNS-server detour cannot work. FakeIP hands the resolution to the
  corp side (mirroring `curl --socks5-hostname`).
- **Trust the corp interception / internal CAs system-wide.** Some corp
  services serve certs the Mac's default trust store rejects (`self signed
  certificate in certificate chain` / `unable to get local issuer
  certificate`) — notably Tinkoff's internal **LBaaS Kubernetes CA** on
  `*.t-tech.team`. Add the relevant corp root CA(s) to the macOS System
  keychain as a trust root so corp HTTPS validates without `--insecure`. See
  design Decision 6 for the specific CA, the trade-off accepted, and why
  scoped per-tool trust was not used.
- Move the WireGuard `private_key` and `pre_shared_key` out of cleartext
  into **sops**; generate `singbox.json` from a nix-managed template.
- Remove the untracked root-level `singbox.json` in favor of the
  flake-managed module and generated config.
- Ensure the Mac runs `openssh` so the corp laptop can dial in.

This is scoped to `fedorivns-mbp`. `fedorivns-thinkpad` and any homelab host
are untouched.

## Capabilities

### New Capabilities

- `corp-proxy-bridge`: The reverse-SSH SOCKS bridge that yields a working
  `127.0.0.1:1080` SOCKS5 endpoint on the Mac whose connections exit via the
  corp laptop. Covers the outbound-dial direction (inbound is blocked),
  stock-`ssh`-only two-hop topology (corp reverse-forwards its sshd; the Mac
  dials back with `-D` so SOCKS egresses from corp), on-demand corp dial vs.
  a persistent loopback-only Mac-side consumer, and the contract that
  sing-box depends on this endpoint.
- `split-tunnel-router`: The sing-box TUN router on `fedorivns-mbp` that
  splits traffic — personal WireGuard by default, `*.tcsbank.ru` via the
  corp bridge — with DNS hijacking, FakeIP for corp domains, private-IP
  direct routing, secrets sourced from sops, and config generated by nix.
- `corp-tls-trust`: System-wide trust of the corp internal/interception root
  CA (macOS System keychain) so corp HTTPS validates from the Mac without
  `--insecure`. Trade-off accepted per design Decision 6.

### Modified Capabilities

None. The listed capabilities are new; no existing `openspec/specs/`
capability changes its requirements.

## Impact

- **Flake structure:**
  - `modules/home/singbox.nix` (already present) renders the personal/corp
    sing-box configs via sops templates.
  - `modules/home/corp-tunnel.nix` (new) — the Mac-side reverse-tunnel consumer
    as a home-manager launchd agent running stock `ssh -D … -p 2222`
    (`KeepAlive`), gated on `me.isMainMachine`.
  - The corp side is a bare on-demand script (`scripts/corp-tunnel.sh`), not
    a managed service.
  - Generated `singbox.json` replaces the untracked root-level file.
- **Secrets:** `secrets.yaml` gains the WireGuard `private_key` and
  `pre_shared_key`; `singbox.json` is templated so no key lands in the Nix
  store world-readable. `.sops.yaml` recipients unchanged.
- **Packages:** `sing-box` on `fedorivns-mbp`. No `autossh` — the Mac-side
  consumer is stock `ssh` supervised by launchd `KeepAlive`; the corp side is
  stock `ssh` too.
- **TLS trust:** the corp internal/interception root CA is exported from the
  corp laptop's keychain and added to the macOS System keychain as a trust
  root (system-wide). Trade-off accepted per design Decision 6.
- **Mac system:** `openssh` enabled (inbound from the corp laptop on the LAN
  only) so the corp laptop can establish the reverse forward (session 1).
- **Corp laptop:** built-in Remote Login (sshd) enabled and the Mac's key
  authorized, so the Mac can dial back over the tunnel (session 2). This is
  the only host-state required on the corp laptop; it is stock macOS, not
  installed software.
- **Existing hosts:** `fedorivns-thinkpad`, homelab — untouched.
- **Manual / external dependency (out of nix-config's reach):** the corp
  laptop is unmanaged; its outbound `ssh -R` dial-in must be initiated on
  that machine (a documented one-liner / script). nix-config manages only
  the Mac side.
- **Non-goals (out of scope):**
  - Automating the corp laptop, or installing third-party software on it.
    (Enabling built-in Remote Login + authorizing the Mac's key and running
    the documented on-demand dial are the only corp-side setup.)
  - Bypassing, disabling, or evading the corp security agents — the design
    explicitly works *with* the outbound-only constraint.
  - Routing any non-`tcsbank.ru` traffic through the corp egress.
  - Multi-host rollout — this is `fedorivns-mbp` only.
