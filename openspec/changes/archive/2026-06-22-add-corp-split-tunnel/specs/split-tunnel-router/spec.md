## ADDED Requirements

### Requirement: Default traffic via personal WireGuard

The router SHALL send all traffic that does not match a more specific rule to
the personal WireGuard endpoint. Non-corp traffic MUST NOT traverse the corp
proxy bridge.

#### Scenario: Generic traffic exits via WireGuard

- **WHEN** the router is running and `curl https://ifconfig.me` is executed on
  `fedorivns-mbp`
- **THEN** the observed egress IP belongs to the WireGuard endpoint's network,
  not the local ISP and not the corp laptop

### Requirement: Corp domains routed via the proxy bridge

The router SHALL route connections to `*.tcsbank.ru` and `*.t-tech.team`
through the `corp-proxy-bridge` SOCKS endpoint (`127.0.0.1:1080`).

#### Scenario: Corp host is reachable through the router

- **WHEN** the bridge is up and `curl https://wiki.tcsbank.ru` or
  `curl https://copilot.t-tech.team` is executed with no explicit proxy
- **THEN** the corp-internal host responds

#### Scenario: Corp connection is tagged to the bridge outbound

- **WHEN** a `*.tcsbank.ru` or `*.t-tech.team` connection is opened
- **THEN** the router logs show it routed to the corp SOCKS outbound, not the
  WireGuard outbound

### Requirement: FakeIP resolution for corp domains

The router SHALL resolve `*.tcsbank.ru` and `*.t-tech.team` using FakeIP,
returning a placeholder address from a reserved range and deferring real name
resolution to the corp laptop via the SOCKS bridge. It MUST NOT depend on
querying the corp resolver directly from the Mac (the corp resolver is
UDP-only and unreachable over the TCP-only SOCKS path).

#### Scenario: Corp domain resolves to a FakeIP

- **WHEN** `fedorivns-mbp` resolves `wiki.tcsbank.ru` or
  `copilot.t-tech.team` while the router is running
- **THEN** the returned address is within the configured FakeIP range
  (`198.18.0.0/15`)

#### Scenario: Real resolution happens at the corp side

- **WHEN** a connection to the FakeIP for a corp domain is initiated
- **THEN** the router passes the original hostname to the SOCKS bridge, and
  the corp laptop performs the actual DNS resolution

### Requirement: DNS is hijacked by the router

The router SHALL intercept system DNS queries so that domain-based routing and
FakeIP take effect for all applications, not only those configured with an
explicit resolver.

#### Scenario: DNS queries are captured

- **WHEN** any application on `fedorivns-mbp` issues a DNS query while the
  router is running
- **THEN** the query is handled by the router's DNS rules rather than going
  directly to an external resolver

### Requirement: Private and non-corp DNS unaffected

The router SHALL route private/RFC1918 destinations directly and SHALL resolve
non-corp domains via the WireGuard-side resolver, leaving normal name
resolution intact.

#### Scenario: Private addresses bypass both tunnels

- **WHEN** traffic is destined for a private (RFC1918) address
- **THEN** it is routed directly, not through WireGuard or the corp bridge

#### Scenario: Non-corp names resolve normally

- **WHEN** a domain outside `tcsbank.ru` and `t-tech.team` is resolved
- **THEN** it is resolved via the default (WireGuard) resolver and is not
  assigned a FakeIP

### Requirement: Configuration generated by nix with sops secrets

The sing-box configuration SHALL be generated from a nix-managed template on
`fedorivns-mbp`, with the WireGuard `private_key` and `pre_shared_key` sourced
from sops. No WireGuard secret SHALL be committed in cleartext or be left
world-readable.

#### Scenario: No cleartext keys in the repo

- **WHEN** the repository tree is inspected after this change
- **THEN** no WireGuard `private_key` or `pre_shared_key` appears in cleartext
  (the untracked root `singbox.json` is gone; keys live in `secrets.yaml`)

#### Scenario: Rendered config is not world-readable

- **WHEN** the generated config containing the decrypted keys exists on the
  activated host
- **THEN** its file permissions restrict read access to the owning user
